## Semantic Versioning (SemVer)
Для найменування версій в файлі `CHANGELOG.md` ми використовуємо принцип [Semantic Versioning](https://semver.org/spec/v2.0.0.html).
<br>
Формат наступний: `MAJOR`.`MINOR`.`PATCH`

- `MAJOR` - Глобальні зміни з втратою сумісності зі старим кодом
- `MINOR` - Новий функціонал без втрати сумісності зі ст арим
- `PATCH` - виправлення багів, невеликі зміни в проекті

## Keep a Changelog
### ! Обово'язково
Найновіша версія **завжди зверху!**
<br>
<br>
Для ведення документації в файлі `CHANGELOG.md` ми використовуємо стандартні заголовки для кожної версії, по принципу [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), а саме:

- `Додано (Added)` - для нових функцій
- `Змінено (Changed)` - для змін в існуючому функціоналі
- `Застаріло (Deprecated)` - для того, що скоро буде змінено / видалено
- `Видалено (Removed)` - для видалення функцій
- `Виправлено (Fixed)` - для виправлення помилок
- `Безпека (Security)` - у випадку покращення безпеки

Також корисно мати розділ `[Unreleased]` у самому верху для змін, які вже зроблені, але ще не отримали нової версії! **(Опціонально)**

## Стандарт для комітів
Ми використовуємо загальний стиль створення комітів, а саме: `тип(область): короткий опис`
<br>
<br>
Список типів:

- `feat`: Додана нова функція
- `fix`: Виправлена помилка
- `docs`: Зміни в документації
- `style`: Зміни / правки у форматуванні (пробіли, коми, табуляція), що не впливають на логіку 
- `refactor`: Правки коду, які не є ні фіксом, ні фічою
- `chore`: Рутинні завдання, не пов'язані з кодом додатку
- `perf`: Правки коду, виключно для покращення швидкодії проекту

## Правила роботи з вітками
Під час реалізації того чи іншого пункту проекту, кожен розробник ПОВИНЕН створювати "локальні гілки" для реалізації певних пунктів поза головною віткою `main`
<br>
<br>
Назви віток повинні бути створенні у наступному форматі: `(тип)/назва`
<br>
Бажано, щоб в назві гілки використовувався нижній регістр та розділенні слів дефісом, наприклад: `feat/add-payment-method`

Типи для назви віток:
- `feat/` (feature) - розробка нової функціональності.
- `fix/` (bugfix) - виправлення помилок, які були знайдені під час розробки або тестування
- `hotfix/` - критичні виправлення, які потрібно внести терміново
- `refactor/` - покращення структури коду без зміни його логіки (чистка "брудного коду, розьиття великих функцій на менші, тощо")
- `docs/` - робота виключно з документацією (оновлення `CHANGELOG.md`, `README.md`, або коментарів у коді)
- `style/` - правки, що стосуються лише зовнішнього вигляду або форматування коду, які не змінюють роботи програми
- `chore/` - рутинні завдання, не пов'язані з кодом додатку: оновлення залежностей (бібліотек), налаштування конфігів, зміна налаштувань збірки проекту, вирішення конфліктів, тощо.
- `perf/` (performance) - зміни, спрямовані виключно на підвищення швидкодії програми
- `experimental/` - коли ви хочете протестувати якусь ідею, але не впевнені, що вона взагалі потрапить у проект 

## Правила роботи з Git
Під час роботи з Git при розробці проекту, треба чітко виконувати наступні правила:
1) Ніхто не має права робити `git push` прямо в гілку `main`. Це може порушити стабільність тієї чи іншою версії проекту. <br><br> Натомість треба використовувати **Pull Request (PR)**, щоб інший розробник міг переглянути код (Code Review) перед злиттям.

2) Гілка має бути "короткоживучою". Як тільки фіча готова і влита в основну гілку, локальну та віддалену гілки фіча треба видаляти, щоб не захаращувати репозиторій.

3) Перед створенням своєї гілки, завжди робіть `git pull origin main`, щоб починати роботу на найактуальнішому коді.  

4) При створенні Pull Request (PR), автор має коротко описати, що було зроблено.<br> - Що змінено?<br>- На що звернути увагу під час перевірки?<br>- (Опціонально) Скріншот, якщо зміни стосуються інтерфейсу

5) Вирішення конфліктів злиття (Merge Conflicts). Ось детальний алгоритм як правильно вирішувати конфлікти злиття та відправляти Pull Request до репозиторію GitHub:

- Крок 1: Оновлюємо локальний **main**.<br>Для початку переходимо до локальної головної гілки: `git checkout main`<br>Забираємо з серверу свіжу версію **main**: `git pull origin main`

- Крок 2: Повертаємось до свого робочого середовища.<br>Повертаємо до гілки, де лежить твій код: `git checkout [назва робочї гілки]`

- Крок 3: Зливаємо оновлену версію **main** до робочої гілки.<br>Даємо команду злити свіжий **main** до твоєї поточної гілки: `git merge main`

- Крок 4: Вирішуємо конфлікти злиття вручну в редакторі коду

- Крок 5: Фіксуємо вирішення.<br>Записуємо всі зміни файлів: `git add .` або `git add [назва файлу]`<br>Створюємо коміт з вирішеним конфліктом злиття: `git commit -m "chore(область): короткий опис"`

- Крок 6: Відправляємо чистий результат на сервер.<br>Пушимо оновлену вітку назад до хмари: `git push origin [назва робочої гілки]`
